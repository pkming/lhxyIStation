# 周末调试前核对单

这份文档只解决一个问题：

- 周末上真机前，基础设置、站点学习、视频监控 / DVR 这三块要先核对什么
- 哪些已经是代码问题，哪些还只是能力没接完
- 真机调试时先看什么，减少来回试错

---

## 1. 使用方式

周末调试前不要直接上手点页面，统一先过这份核对单。

顺序固定：

1. 先看“当前代码已承接到哪”
2. 再看“旧项目关键行为是什么”
3. 再看“本次真机要验证什么”
4. 最后看“哪些不是这次要当场修的”

---

## 2. 基础设置

### 2.1 当前代码已承接到哪

当前新壳承接点：

1. `LegacyBasicSetupActivity`
2. `LegacyBasicSetupSectionFragment`
3. `ShellConfig / ShellConfigRepository / ShellRuntime`

当前已经明确接上的行为：

1. 各子页旧布局切换
2. 第一批字段回显
3. 第一批运行期配置保存
4. 第一批确认弹窗
5. 语言 / 网络 / 串口保存后的重启语义
6. 无线页跳系统设置

### 2.2 旧项目关键行为

旧项目关键点不是只有保存值，还包括：

1. 资源导入前置判断
2. 网络和串口协议互斥
3. 部分保存后重启
4. 广告授权相关分支
5. 串口真实下发和设备联动

### 2.3 当前已知缺口

1. 资源导入状态源已收口到运行期配置，文件管理页现在会真实扫描目录、解压 ZIP 资源并写回这个状态
2. 网络/串口协议互斥已收口到运行期配置里的 dispatchOwner，首页、调度模块和报站模块都已开始消费这个状态
3. 当 dispatchOwner=serial_rs232_1 且 RS232-1Protocol=DVR 时，确认公告、确认调度、发车、GPS 上报、站点信息上报、司机签到签退都已开始走真实串口发送；其余串口业务还未补齐
4. 广告授权分支未恢复

### 2.4 周末真机优先验证

优先只验证下面这些：

1. 保存后运行期配置是否真的生效
2. 重启请求在真机 systemOps 下是否成功
3. 串口波特率和协议选择是否影响真实设备
4. 网络 host / port / 默认调度通道是否按配置走
5. 无线页系统设置入口是否正常打开

### 2.5 调试前逻辑结论

这块周末前不需要再大改结构，重点是：

1. 带着“资源导入状态已持久化、ZIP 导入链路已接，但 rar/旧数据库落库未恢复”和“dispatchOwner 已落配置，DVR 串口调度已接第一批真实发帧，但串口业务仍未全量恢复”的预期去调
2. 先验证已接通能力，不把未完成能力当成现场 bug

## 3. 站点学习

### 3.1 当前代码已承接到哪

当前新壳承接点：

1. `LegacySiteCollectionActivity`
2. `LegacySiteCollectionSectionFragment`
3. `StationBusinessModule`
4. `GpsSerialMonitor`

当前已经明确接上的行为：

1. 旧布局壳
2. 方向切换
3. 示例线路 / 示例站点或属性切换
4. GPS 当前值展示
5. 点击保存后推进到下一项

### 3.2 旧项目关键行为

旧项目真正的流程是：

1. 线路滚轮选择
2. 方向切换后重新拉站点或属性列表
3. 站点 / 属性滚轮选择
4. GPS 串口实时刷新当前值
5. 确认后把当前 GPS 覆盖写回对应数据项
6. 自动推进到下一项
7. 到末项后切换上下行并继续

### 3.3 当前已知缺口

1. 当前线路和属性是示例数据，不是旧数据库真实列表
2. 当前保存动作是挂在 `station.advance_station` 上，不是真实采集落库
3. 当前没有恢复旧项目滚轮选择器
4. 当前没有恢复旧项目末项切换上下行后的真实数据流

### 3.4 周末真机优先验证

这块周末更适合只验证：

1. GPS 串口监听是否稳定
2. 当前页面 GPS 数据显示是否正确
3. 方向切换后的页面状态是否正确
4. 保存操作是否至少能稳定触发页面推进和日志输出

不建议周末现场把它当成“真实采集流已经恢复”来验证。

### 3.5 调试前逻辑结论

这块当前状态要定义成：

1. 页面骨架和交互路径已恢复
2. GPS 展示路径已恢复
3. 真实学习落库流程未恢复

周末调试时要避免把“缺数据库和缺滚轮选择器”误判成真机问题。

## 4. 视频监控 / DVR

### 4.1 当前代码已承接到哪

当前新壳承接点：

1. `LegacyVideoMonitorActivity`
2. `CameraDvrBusinessModule`
3. `CameraAdapter / GpioAdapter`

当前已经明确接上的行为：

1. 旧页面壳
2. 默认 Camera 打开 / 关闭链路
3. 页面打开时尝试开 Camera
4. 页面退出时尝试关 Camera

### 4.2 旧项目关键行为

旧项目这里除了 Camera 预览，还有：

1. DVR 键位 -> 串口协议码映射发送
2. 摄像头 GPIO 开关
3. 指定 cameraId 打开预览
4. 触摸区坐标 -> DVR 串口触摸帧发送

### 4.3 当前已知缺口

1. 当前 DVR 触摸协议已接第一版，按 M90 `1280x800` 坐标系缩放并发送触摸坐标帧，坐标字节序仍需真机核对
2. 当前 DVR 键位已按旧键码映射恢复到第一批真实串口发送

### 4.4 周末真机优先验证

这块周末只优先验证：

1. 默认 Camera 打开是否成功
2. 页面切入切出时 Camera 开关是否稳定
3. GPIO 与 Camera 通道映射是否正确
4. 预览区触摸是否同时产出 UI 坐标日志和 `DVR_TOUCH_EVENT` 协议日志

可以把 DVR 键位协议纳入本轮已完成能力去验；触摸协议也可以纳入联调范围，但当前应按“已接第一版、重点核对坐标/字节序”的口径去验。

触摸协议本轮也可以开始联调，但要按“已接第一版、重点核对坐标/字节序”的预期去验。

### 4.4.1 触摸联调固定动作

周末现场至少固定做这 5 步：

1. 进入视频监控页后，先确认默认 Camera 打开成功。
2. 在预览区左上角点击一次，确认 UI 日志里有 `phase=down/up`、raw、view、scaled。
3. 在预览区中心点击一次，确认协议日志里出现 `DVR_TOUCH_EVENT`，并能直接看到 x/y/checksum。
4. 在预览区右下角点击一次，确认坐标明显大于左上和中心，避免整屏都发同一组值。
5. 如果设备无响应，先记录 3 组日志再判断是坐标系、字节序还是设备接线，不要现场回到页面层乱改。

### 4.5 调试前逻辑结论

这块当前状态要定义成：

1. Camera 开关链路已恢复到正式入口
2. DVR 键位协议已恢复到正式串口发送
3. DVR 触摸协议已恢复到第一版串口发送，当前重点风险在坐标系和字节序

周末现场如果 DVR 键或触摸无真实效果，都应先按“已接能力异常”处理，再判断是协议值、坐标换算还是接线问题。

触摸这块的判定顺序固定为：

1. 没有 UI 触摸日志：先查触摸采集或控件层。
2. 有 UI 触摸日志但没有 `DVR_TOUCH_EVENT`：先查模块动作或串口发送入口。
3. 有 `DVR_TOUCH_EVENT` 但 DVR 无效果：优先查坐标系、字节序、接线或设备协议差异。

## 5. 周末前统一自检

周末前统一再过一次下面这些：

1. 这三块文档里的“当前已知缺口”有没有新变化
2. 页面入口有没有改动
3. 配置项有没有改动
4. 日志点够不够看问题
5. 哪些是现场只验证、不现场改结构

---

## 6. 周末调试顺序建议

建议顺序：

1. 基础设置
2. 站点学习
3. 视频监控 / DVR

原因：

1. 基础设置最容易直接验证配置是否生效
2. 站点学习主要看 GPS 和页面状态
3. 视频 / DVR 最依赖真机映射和协议，放最后更稳

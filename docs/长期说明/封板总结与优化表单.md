# 封板总结与优化表单

这份文档只做两件事：

1. 说明当前为什么可以进入代码封板。
2. 把当前做得不够好的地方、后续可优化项和是否影响封板，集中列成一张表。

---

## 1. 当前结论

当前可以按“代码封板，转真机联调/回归”推进。

原因：

1. 主页面壳层、主链入口、下载链首页提示、首页方向/速度/摘要区、视频监控页等待层与触摸路径，已经都按 `M90` 口径收口到正式承接点。
2. 当前剩余的大头，已经不是“页面没迁完”，而是硬件映射、真实数据源、真机观感和联调证据。
3. 如果继续在没有新证据的前提下盲改代码，收益很低，反而容易把已经对齐的 `M90` 口径再改偏。

当前不要误说成“100% 全部完成”。

保留这个边界的原因：

1. 首页 `FIN/FOUT/BIN/BOUT/ALL` 对应的 `JHY` 客流源仍未接入。
2. 视频相关仍需真机确认触摸字节序、GPIO pin 和等待层观感。
3. 部分 GPS/站点超速项仍受资源字段缺失约束。
4. `100% 复刻` 的正式判定口径，统一以 [../清单/M90对照迁移清单.md](../清单/M90对照迁移清单.md) 里的“什么情况下才能说 100% 复刻 M90”为准，不再靠会话口头解释。

## 3. 做得不够好的地方和优化表单

| 项目 | 当前做得不够好的地方 | 影响 | 建议优化 | 优先级 | 是否影响当前封板 |
| --- | --- | --- | --- | --- | --- |
| 文档入口 | 基线、封板口径和主入口之前分散在清单、差异文档、会话记录里 | 新人或新会话容易先看错文档，重复判断一次“还能不能继续补代码” | 固定把 `M90对照迁移清单`、`新壳与M90差异记录`、本表单作为三份主入口 | 高 | 否 |
| 状态口径 | 之前一度把“待验证”和“代码未闭合”混在一起写 | 容易误判是否还要继续开发 | 后续所有条目都按 `未开始 / 进行中 / 待验证 / 已闭合` 严格区分 | 高 | 否 |
| 视频页收口 | 视频监控页中途保留过一组不属于 `M90` 正式页的额外控件 | 会把真机联调带偏，用户以为页面还没复刻对 | 后续视频专项只围绕 `等待层 / 预览 / 返回浮层 / 触摸 / 自动切换` 五块回归 | 高 | 否 |
| 无源字段处理 | 首页客流计数条最初用 `0` 做稳定占位，不够贴近 `M90` 无源态 | 会让人误以为已有真实人数统计 | 对所有“无真实来源”的字段，统一标清是 `N`、空串还是配置回退，禁止伪造业务值 | 高 | 否 |
| 真机证据沉淀 | 很多结论只在会话里说过，没第一时间沉淀成长期文档 | 后续容易重复考古、重复试错 | 真机回归后，把 pin、触摸字节序、联调日志样例直接补进专项调试文档 | 高 | 否 |
| 历史文档干扰 | 仓库里仍保留大量旧 `4.4 / K80` 文档和章节标题 | 搜索时容易命中旧标准，造成误导 | 历史文档继续保留，但在入口文档中统一注明“只作考古，不作当前完成标准” | 中 | 否 |
| 首页专项验证 | 首页条目很多已经到 `待验证`，但缺一张按按钮和区域拆开的最终回归表 | 真机回归时容易漏点 | 后续把首页回归结果补录到 `首页完整核对清单.md`，按按钮/状态区/视频区分段记录 | 中 | 否 |
| GPS 资源依赖 | 站点超速 20/21 等链路受本地 CSV 字段限制，当前文档里说明分散 | 容易把资源缺失误判成代码未实现 | 后续把“资源不足导致无法闭合”的点统一归档到 `暂缓与待确认问题归档.md` | 中 | 否 |

## 4. 封板后建议执行顺序

1. 先按 [../清单/联调前1-N核对清单.md](../清单/联调前1-N核对清单.md) 做联调前准备。
2. 再按 [../清单/周末调试前核对单.md](../清单/周末调试前核对单.md) 和首页/视频专项清单逐项回归。
3. 真机发现新问题后，先判断是 `待验证` 项落空，还是出现了新的明确代码缺口。
4. 只有拿到新证据后，才重新打开代码修改。

---

## 5. 当前封板边界

当前可以停改代码，但不要越过下面这些边界：

1. 不宣称“100% 全部完成”。
2. 不把硬件联调问题重新表述成页面壳层未复刻。
3. 不在没有 `M90` 证据和真实来源的前提下继续补新语义。
4. 不把 `4.4 / K80` 历史资料重新抬成第二套现行标准。
5. 不在主单仍有 `待验证 / 待确认 / 资源缺失 / 协议未知` 条目的情况下提前写“100% 复刻”。

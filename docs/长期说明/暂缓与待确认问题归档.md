# 暂缓与待确认问题归档

这份文档专门归档一类问题：

1. 当前先不做，但后面很容易被重新提起。
2. 旧版有，但新壳暂时缺硬件映射、权限、协议证据或配置承接点，不能硬接。
3. 旧版没有，这一轮迁移明确不纳入范围，避免后面会话又被误当成“遗漏功能”。

如果只是想看设备引脚和旧代码行为对照，先看 [旧终端与M90硬件映射对照表.md](旧终端与M90硬件映射对照表.md)。

---

## 1. 归档规则

### 1.1 旧版没有的功能

统一按下面处理：

1. 当前迁移阶段先不做。
2. 不把“新平台支持”自动等同于“这次要补”。
3. 只有在用户明确改口、要新增能力时，才从这里移出并进入推进路线。

### 1.2 旧版有，但当前证据不闭合的功能

统一按下面处理：

1. 先归档，不硬编码猜测实现。
2. 必须写清楚卡点属于哪一种：
   - 硬件映射不明
   - 协议证据不足
   - Android 权限缺失
   - 运行时配置没承接
   - 真机行为未验证
3. 必须写清楚“下一步要拿什么证据”才能继续做。

### 1.3 已接第一版但不能算完全闭环的能力

统一按下面处理：

1. 可以继续保留已实现部分。
2. 但要显式标注“待真机验证”或“待协议验证”。
3. 不能因为代码已经能编译，就从归档里删掉。

## 3. 归档更新方式

后面再遇到类似问题，统一按这个格式补：

1. 先写“这是不是旧版已有能力”。
2. 再写“当前为什么不做”或“为什么只能做到一半”。
3. 最后写“继续推进前缺什么证据”。

不要只写：

1. `待确认`
2. `后面再说`
3. `真机再看`

必须把卡点写具体，否则下一次会话还会重复判断一遍。

---

## 4. 什么时候可以从这里移出

只有下面两种情况可以移出归档：

1. 已明确决定不做，并且后面也不再作为迁移目标。
2. 所需证据已经补齐，并且已经进入正式推进文档或代码实现。

如果只是“准备做了”或者“已经讨论过”，还不能移出。

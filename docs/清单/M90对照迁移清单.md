# M90 对照迁移清单

这份文档只做一件事：

- 以 M90 成品 APK 为当前新机基线，逐项对照 Android11 新壳还缺什么

旧文档继续保留，但只作为历史参考，不再作为当前完成标准：

1. [UI与业务核对清单.md](UI与业务核对清单.md)
   - 保留历史页面总量和原始核对记录
2. [首页完整核对清单.md](首页完整核对清单.md)
   - 保留首页专项历史闭合项和旧回归记录

这份新清单的区别是：

1. 基线改成 M90 已完成实现，不再默认按历史 `4.4` 推断
2. 每一项都要求同时写清 `M90 对照点`、`Android11 承接点`、`当前状态`、`下一步`
3. 后续迁移按这份单子逐项推进，不再一边翻历史一边临时定范围

## 2. 状态标记

统一只用下面四类：

1. `未开始`
   - 还没承接到 Android11
2. `进行中`
   - 已有第一版，但 UI、行为或协议还没完全对齐 M90
3. `待验证`
   - 代码已承接，需要点击回归、联调或真机确认
4. `已闭合`
   - M90 对照、代码承接和验证口径都已明确

## 3. 执行顺序

后面每次继续迁移，都固定按这个顺序：

1. 首页
2. 登录页
3. 菜单页
4. 线路选择
5. 调度中心
6. 签到/考勤
7. 视频监控
8. 文件管理
9. 基础设置
10. 系统信息 / 信息浏览 / 密码设置
11. 协议和硬件专项回归

原则：

1. 先把正式主链走通，再补外围页
2. 先收 UI 结构和按钮语义，再补深业务细节
3. 一项没有写清 M90 对照点和 Android11 承接点，不能直接标完成

### 3.1 不漏项工作法

后面按 M90 迁移时，不要只盯 UI，也不要只盯协议；统一按“5 本账”推进。

每到一个页面或模块，都必须把下面 5 本账至少过一遍：

1. 页面账
   - 这页在 M90 上到底有哪些区域、按钮、文案、显隐和跳转
   - 先解决“看起来是不是同一页”
2. 动作账
   - 这页每个按钮点下去做什么，顺序是什么，回到哪里
   - 先解决“能不能按旧流程走完”
3. 状态账
   - 这页每个字段的数据从哪里来，谁负责刷新，什么条件下变化
   - 先解决“显示是不是来自真实状态，不是占位”
4. 协议账
   - 这页背后是否依赖 GPS、DVR、串口、调度、签到、外设事件
   - 先解决“页面动作后，底层该不该发帧、收状态、写配置”
5. 验证账
   - 这页当前到底做了什么验证：静态对照、点击回归、日志验证、真机验证
   - 先解决“当前结论有没有证据支撑”

缺任意一本账，都不能标 `已闭合`。

### 3.2 每次迁移固定顺序

每次继续某一页时，都固定按下面顺序，不要跳着做：

1. 先列 M90 对照点
2. 再列 Android11 承接点
3. 先补页面账和动作账
4. 再补状态账和协议账
5. 最后补验证账和结论

这样做的目的只有一个：

1. 避免只把页面画出来，业务没接上
2. 避免只把协议补出来，页面没有正式入口
3. 避免做完一轮之后说不清当前到底验证到哪

### 3.3 页面完成门槛

后面任一页面或模块，只有同时满足下面几条，才能从 `进行中/待验证` 往 `已闭合` 走：

1. 页面结构已和 M90 对上
2. 主要按钮语义和回收链已对上
3. 关键显示字段已经能落到真实状态源
4. 相关底层依赖已经明确
   - 不要求一次全做完，但必须写清“已接上”“未接上”还是“待真机确认”
5. 至少有一轮可复述的验证结论
   - 例如点击回归、日志验证、真机验证之一

如果某页当前只能做到 UI 一致，也可以继续推进，但状态只能写 `进行中`，不能提前写 `已闭合`。

### 3.4 跨页专项怎么防漏

像 GPS、DVR、签到、调度、串口这类不是单页独占的链路，不要混在页面备注里带过，统一双处记录：

1. 在页面表里记“这页依赖了什么专项”
2. 在专项表里记“这个专项影响了哪些页面”

最少要保证下面这条对应关系能被追出来：

1. 首页状态栏和速度区 <- GPS
2. 视频监控页、首页视频相关动作 <- DVR
3. 首页司机区、调度中心考勤 <- 签到/考勤
4. 调度中心、首页提示区、计划时间 <- 调度链
5. 基础设置里的网络/串口/协议选择 <- 配置与串口链

这样后面即使先做 UI，也不会把底层依赖做丢。

## 4. 首页对照迁移表

当前首页是第一优先级，先按下面的表推进。

### 4.1 首页 5 本账示例

后面如果不知道首页这一页该怎么按“5 本账”落地，就先照这个示例走。

#### 4.1.1 页面账

首页先固定看这几块，不要一上来就按零碎控件散着改：

1. 顶部状态栏
2. 左侧固定按钮列
3. 数字键盘区
4. 中间主业务区
5. 右侧司机/提示/视频区

首页页面账先回答这几个问题：

1. M90 首页到底分成哪些区域
2. 哪些区域是固定显示，哪些是条件显示
3. Android11 当前布局已经对上哪些，哪些还只是第一版近似

#### 4.1.2 动作账

首页动作账不要漏下面这几组：

1. 左侧固定按钮列
   - 报站
   - 重复
   - 切换
   - 键盘
   - 停止
   - 第六键语义
   - 司机/退出键
   - 菜单
2. 数字键盘区
   - `0-9`
   - `↑`
   - `↓`
   - `消息查看`
   - `排班查询`
   - `ESC`
3. 首页主链
   - 首页 -> 登录 -> 菜单 -> 子页 -> 首页

首页动作账先回答这几个问题：

1. 每个按钮在 M90 上做什么
2. Android11 当前接到了哪个 action 或页面
3. 当前是“旧 4.4 语义”“M90 语义”还是“还没定稿”

#### 4.1.3 状态账

首页状态账至少盯住这些字段，不要只看按钮：

1. 时间、卫星、LAN、CMS、DVR、4G、WiFi、GPS
2. 线路名、终点站、线路摘要
3. 当前站 / 下一站标签与名称
4. 司机信息、提示区、计划时间
5. 速度 / 限速

首页状态账先回答这几个问题：

1. 每个字段当前来自哪个状态源
2. 哪些字段是实时刷新，哪些字段是动作后刷新
3. 哪些字段现在还是占位或近似值

#### 4.1.4 协议账

首页协议账至少要把下面几条依赖写明：

1. 顶部 GPS、卫星、速度区 <- GPS 链
2. 首页提示区、计划时间、CMS <- 调度链
3. 司机区和司机动作 <- 签到/考勤链
4. 数字键盘 `0-9` <- 报站服务音 + RS485 外发链
5. 第六键、视频区、相关首页动作 <- DVR / 串口链

首页协议账先回答这几个问题：

1. 当前动作后有没有真实发帧或状态写回
2. 哪些专项已经接上第一版
3. 哪些地方还只能做页面占位，不能假装底层已经闭合

#### 4.1.5 验证账

首页验证账统一只记下面这几类，不要写模糊结论：

1. 静态对照
   - 对照 M90 布局、代码、旧首页路径
2. 点击回归
   - 首页按钮、数字键盘、首页主链
3. 日志验证
   - 动作后状态刷新、模块执行、串口发帧
4. 真机验证
   - GPS、DVR、串口、系统设置跳转等硬件相关能力

首页验证账当前最少要补：

1. 首页主链点击回归
2. 左侧按钮列点击回归
3. 数字键盘点击回归
4. GPS / DVR 相关字段的日志或真机结论

#### 4.1.6 首页这一页怎么判定阶段

首页后面每次推进时，统一按下面判断，不要凭感觉写状态：

1. 只改了视觉和布局
   - 记 `进行中`
2. 动作已接上，但还没完成点击回归
   - 记 `待验证`
3. 页面、动作、状态、协议、验证 5 本账都已经能闭环复述
   - 才能记 `已闭合`

### 4.2 首页当前优先补齐项

   - `0-9`、`↑`、`↓`、`ESC` 先逐个补点击结论

   - 明确哪些能接真实状态，哪些先占位

# 首页完整核对清单

这份文档只解决一个问题：

- 首页后面到底按什么顺序继续收
- 哪些已经闭合，哪些只是第一版，哪些还没开始
- 每次继续首页时先核对什么，避免又回到“边翻历史边猜下一步”

如果当前还没接上最近页面回归修补，先看 [../归档/会话记录/新会话交接-2026-05-06.md](../归档/会话记录/新会话交接-2026-05-06.md)。

如果这次是准备做真机前全链路核对，先看 [联调前1-N核对清单.md](联调前1-N核对清单.md)。

---

## 1. 使用方式

这份清单不是背景说明，而是首页专项执行单。

每次继续首页时都按同一顺序过，不要跳着做：

1. 先核对入口和当前基线。
2. 再核对顶部状态栏、左侧按钮列、主业务区、数字键盘四个大块。
3. 再核对首页和登录、菜单、线路选择、站点学习之间的联动。
4. 最后才决定这次是做点击回归、UI 细节修补，还是业务语义补齐。

每一项统一按下面三类判断：

1. `已闭合`
   - 当前代码和文档口径已经对齐，下一次只需要回归，不要重复设计。
2. `第一版待确认`
   - 当前已经接上，但还需要点击回归或真机核对细节。
3. `待补齐`
   - 当前还没恢复到旧版语义，后续继续按这份单子推进。

## 3. 首页完成标准

首页后面什么时候可以不再卡在这里，统一按下面判断：

1. 首页 -> 登录 -> 菜单 -> 子页 -> 首页回收链点击回归稳定。
2. 顶部状态栏字段来源、显示节奏和刷新口径明确。
3. 左侧固定按钮每个旧动作都已经有文档口径、代码承接和页面反馈。
4. 数字键盘每个键都能明确归到“已恢复旧行为”“旧版保留键位”或“当前明确不做”。
5. 主业务区关键状态和按钮动作联动关系明确，不再靠人工猜当前应该显示什么。
6. 首页剩余未恢复项能被明确归档成待补齐，不再和已完成项混在一起。

只要这六条还没同时满足，就继续按这份单子推进，不再临时换题。

## 7. 代码注释策略

这些内容不需要整份清单逐条搬进代码注释。

代码注释只落在“迁移决策点”和“看代码本身不容易看懂旧语义”的地方，避免把文件注成流水账。

适合落代码注释的点：

1. 旧语义和新实现明显不一一对应的地方。
   - 例如司机键为什么在“已刷卡但未拿到真实司机身份”时仍显示 `登陆`，而不是直接显示占位司机名。
2. 当前故意保留旧版保留键位的地方。
   - 例如“消息查看”“排班查询”为什么现在不跳页。
3. 当前实现是旧语义近似实现，而不是完全等价实现的地方。
   - 例如 `↑ / ↓` 预览语义最终定稿后，如果实现仍和旧版状态机不完全同构，就值得写注释。
4. 状态来源容易被误判的地方。
   - 例如 CMS、GPS、DVR 状态是怎么从状态对象或设备状态推导出来的。
5. 页面只负责展示，但看起来像在做业务判断的地方。
   - 例如 `refreshHomeState()` 里哪些只是展示映射，哪些是为了保持旧首页显示语义。

不需要落代码注释的点：

1. 纯直译 UI 赋值。
2. 明显的按钮点击跳转。
3. 和旧实现一一对应、代码本身已经足够直接的简单分支。

当前更建议先把注释放在这几类代码点上，而不是全页铺开：

1. `LegacyMainActivity.bindActions()`
2. `LegacyMainActivity.quickPreviewForward()` / `quickPreviewBackward()`
3. `LegacyMainActivity.refreshHomeState()`
4. `LegacyMainActivity.updateLineChoiceShortcutButton()`

## 8. 后续推进顺序

首页后面统一按这个顺序继续，不再反复换：

1. 先跑一轮点击级回归。
   - 首页 -> 登录 -> 菜单 -> 子页 -> 首页
   - 左侧按钮列
   - 数字键盘区
2. 再收最容易反复卡住的两块。
   - `↑ / ↓` 预览语义
   - 主业务区字段联动
3. 再收顶部状态栏细节。
   - 时钟刷新
   - CMS / DVR / GPS 状态节奏
4. 最后补首页相关公共弹窗和视觉细节。

当前不建议反过来先做：

1. 不先重写首页结构。
2. 不先把旧项目所有隐式逻辑都塞回页面层。
3. 不在 `↑ / ↓` 和主业务区口径没定前继续加首页新动作。

# 推进步骤

如果当前只是想先知道“这次该看哪份文档”，先回到 [README.md](README.md)。

这份文档只解决一个问题：

- 后面到底从哪里开始
- 先做什么
- 后做什么
- 什么不能提前做

后面推进统一按这份走，不再边做边换方向。

---

## 1. 当前总方向

### 1.1 目标

#### 1.1.1

这次不是做一套新 UI，而是：

1. 旧 `4.4` UI 壳还原
2. 新壳逻辑接进去
3. 再按旧逻辑逐项补齐

#### 1.1.2

一句话就是：

`正式 UI 用旧的，逻辑内核用新的，debug 能力单独保留。`

#### 1.1.3 兼容原则

后面统一按这个口径推进：

1. 业务基线以旧 `4.4 / K80` 为准。
2. 新项目和 `Android 11 / M90` 要向这套旧业务结果兼容。
3. 不是旧 `4.4` 去“向前兼容”新实现。
4. 任何兼容方案都不能以明显提高系统复杂度为代价。

一句话就是：

`旧业务往后兼容到新平台，不做旧平台向前兼容，不为了兼容继续加复杂度。`

### 1.2 当前不要做的事

#### 1.2.1

下面这些现在不要继续扩：

1. 不再继续设计新正式页面
2. 不再继续把 `ModuleCenterActivity` 当最终正式 UI 扩下去
3. 不再先做协议细节联调
4. 不再先做真机差异修补

这些都要排在`旧 UI 壳还原`之后。

#### 1.2.2 现在不要做的兼容方向

下面这些方向默认不要走：

1. 不为了照顾旧 `4.4` 历史实现，把新架构重新打散回页面层。
2. 不为了“两个平台都像原来一样写”，把同一份业务拆成两套并行实现。
3. 不为了兼容把临时判断、特例分支、双写状态不断堆高。
4. 不把“兼容”理解成新旧双向互相适配；默认只做旧业务向新平台落地。

完成判断标准也统一成：

1. 业务结果更接近旧 `4.4`。
2. 新平台承载能力更稳定。
3. 代码复杂度不上升，或者至少不上升到影响后续维护。

## 2. 推进总顺序

### 2.0 本周范围

#### 2.0.1 本周只做什么

这周统一按“工作日离线推进 + 周末真机收口”走，不再边做边扩范围。

工作日只做：

1. 调度、报站、签到三条主链继续补旧行为和新逻辑接入
2. 文件管理 / 基础设置 / 视频监控这三块继续补离线能闭环的行为
3. 差距台账、日志点、验证步骤、文档同步

周末真机只做：

1. DVR 键位 / 触摸 / Camera / GPIO
2. GPS 串口与配置生效
3. SourceFile.zip 导入结果
4. 保存后重启和系统能力验证

#### 2.0.2 本周不要做什么

1. 不再新开大块页面范围
2. 不为了 M90 单独做第二套业务逻辑
3. 不把真机阶段的问题留到工作日靠猜处理
4. 不做没有文档依据的随机修补

### 2.1 第零阶段

#### 2.1.1 旧版基线固化

先把旧版留档，不然后面“UI 一样”没参照。

##### 2.1.1.1 要留的东西

1. 首页截图
2. 登录页截图
3. 菜单页截图
4. 调度中心截图
5. 报站相关页面截图
6. 签到 / 考勤页截图
7. 视频页截图
8. 文件页截图
9. 基础设置 / 系统信息页截图
10. 关键主流程录屏
11. 关键按钮点击后的反馈记录
12. 关键弹窗样式

##### 2.1.1.2 完成标准

后面任何一个页面对齐时，都能拿出旧版截图或录屏做对照。

### 2.2 第一阶段

#### 2.2.1 旧 UI 资源迁移

先迁壳，不先接深逻辑。

##### 2.2.1.1 要迁的资源

1. `layout`
2. `drawable`
3. `anim`
4. `menu`
5. 旧页面图片资源
6. `strings / colors / dimens / styles`
7. 旧页面相关自定义控件
8. 旧页面相关对话框布局

##### 2.2.1.2 注意点

不是直接整包复制完就结束，还要检查：

1. 资源命名冲突
2. AndroidX / support 差异
3. 主题差异
4. 自定义控件依赖
5. 旧布局里引用的图片和样式是否齐全

##### 2.2.1.3 完成标准

旧 XML 和资源在新壳工程里能编译通过，至少能被正式页面引用。

### 2.3 第二阶段

#### 2.3.1 公共 UI 单元还原

先还原会影响所有页面的公共件。

##### 2.3.1.1 要先做的公共件

1. 顶部状态栏
2. 首页左侧固定按钮列
3. 首页数字键盘区
4. 公共弹窗
5. 公共选择器
6. 菜单风格
7. 公共按钮、输入框、标题栏样式

##### 2.3.1.2 完成标准

后面新建正式页面时，不需要再重复造样式和公共控件。

### 2.4 第三阶段

#### 2.4.1 正式页面 1:1 建立

旧项目一个正式页面，新壳就建立一个正式页面，不再用“一个通用页顶多个旧页面”。

##### 2.4.1.1 第一批正式页面

1. 首页
2. 登录页
3. 菜单页

##### 2.4.1.2 第二批正式页面

1. 调度中心
2. 线路选择
3. 文件管理
4. 视频监控
5. 信息浏览
6. 语音通话

##### 2.4.1.3 第三批正式页面

1. 基础设置
2. 系统信息
3. 站点学习

##### 2.4.1.4 完成标准

旧项目里的正式页面，在新壳里都有对应正式页面骨架。

### 2.5 第四阶段

#### 2.5.1 跳转链恢复

页面有了，还要把跳转恢复成旧版路径。

##### 2.5.1.1 先恢复主链

1. 首页 -> 登录页
2. 登录页 -> 菜单页
3. 菜单页 -> 八个子页面

##### 2.5.1.2 再恢复子链

1. 调度中心里的 tab / fragment 切换
2. 基础设置里的 fragment 切换
3. 系统信息里的 fragment 切换
4. 站点学习里的 fragment 切换

##### 2.5.1.3 完成标准

正式路径的页面跳转顺序和旧版一致。

### 2.6 第五阶段

#### 2.6.1 接新逻辑

这一步才开始把现在的新壳逻辑真正挂到旧 UI 上。

##### 2.6.1.1 接入原则

1. 页面只负责展示和点击
2. 业务状态走新壳状态对象
3. 业务动作走新壳模块 action
4. 设备调用走 `device-api`
5. 协议调用走 `protocol`

##### 2.6.1.2 优先接的三条主链

1. 调度
2. 报站
3. 签到

##### 2.6.1.3 第二批再接

1. 视频 / DVR
2. 升级
3. 文件

##### 2.6.1.4 完成标准

正式页面已经不只是长得像，而是能真正驱动新逻辑。

### 2.7 第六阶段

#### 2.7.1 对照旧逻辑补齐

这一步开始做“是不是完整”的核对。

##### 2.7.1.1 每个页面都按这 5 项核

1. 页面长得是不是一样
2. 跳转是不是一样
3. 按钮动作是不是一样
4. 页面反馈是不是一样
5. 背后联动是不是一样

##### 2.7.1.2 每个业务模块都按这 4 项核

1. 入口是不是一样
2. 状态是不是一样
3. 动作是不是一样
4. 异常处理是不是一样

### 2.8 第七阶段

#### 2.8.1 协议和联调

这一步最后做。

##### 2.8.1.1 包括

1. 串口口位确认
2. GPS 真口
3. RS485 真口
4. DVR 串口和触摸
5. RFID 真适配
6. 升级和系统权限
7. 真机联调和差异修正

## 3. 从哪里开始

### 3.1 当前真正的起手顺序

#### 3.1.1 第一步

这份文档里的前五步已经不再是“待开始”，而是大部分已经落地。

当前实际进度已经走到：

1. 首批旧 UI 资源已迁入
2. 首页 / 登录 / 菜单主链已恢复
3. 第一批子页面已建起来
4. 调度中心 / 系统信息 / 系统设置 / 站点学习右侧内容区都已经开始接首批真实逻辑
5. 视频监控正式页已进入正式路径

所以新会话不要从“再建首页骨架”开始，而是直接从下面几块继续：

1. 首页细项
2. 视频监控细行为
3. 调度中心细动作
4. 系统设置真实持久化入口

#### 3.1.2 如果后面继续补一批新正式页，仍按这个子顺序

先迁旧 UI 资源，不先接深逻辑。

#### 3.1.3 先建骨架

把正式页面 1:1 建出来，先从下面三个开始：

1. 首页
2. 登录页
3. 菜单页

#### 3.1.4 再接跳转

恢复跳转主链：

1. 首页 -> 登录
2. 登录 -> 菜单
3. 菜单 -> 子页面

#### 3.1.5 最后挂主链逻辑

先接三条主链：

1. 调度
2. 报站
3. 签到

1. 旧 UI 壳还原
2. 跳转恢复
3. 新逻辑接入
4. 旧逻辑对照补齐

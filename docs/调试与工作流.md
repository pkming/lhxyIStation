# 调试与工作流

## 1. 这份文档解决什么问题

这个项目不是普通页面项目，真正难的是：

- 串口调试麻烦
- 报文多，肉眼不好追
- 真机问题很容易复现不稳定
- 现场问题如果没留好日志，回来基本没法还原

所以调试和工作流必须单独立规矩。

## 2. 推荐工作流

每个功能按下面顺序推进：

1. 先补文档
   - 写清楚这次要动哪个模块、哪个协议、哪些日志点、哪些配置项。
2. 先做纯逻辑
   - 先把模型、协议、状态机写好。
3. 再做 fake 调试
   - 用假数据或回放数据走流程。
4. 最后接真机
   - 真机只做口位、时序、权限、驱动确认。
5. 收尾时补记录
   - 把踩坑点、映射关系、日志位置写回文档。

## 3. 串口相关的调试顺序

串口不要一上来就拿整条业务链排查，建议分四步：

1. 先确认口位
   - 这个口到底是不是要找的设备。
2. 再确认权限
   - 应用能不能真的打开这个口。
3. 再确认波特率和收发
   - 先看有没有字节流，再谈协议。
4. 最后确认协议解释
   - 有流但不通，才去看编解码和业务状态。

## 4. 日志怎么打

当前这版先按这 6 类记：

- `UI`
- `BIZ`
- `PROTOCOL_TX`
- `PROTOCOL_RX`
- `DEVICE`
- `ERROR`

串口打开、socket 连接、GPIO 状态这些先记到 `DEVICE`。
真实 socket 收到的回包统一先记到 `PROTOCOL_RX`。

建议每条日志都带：

- 时间
- 线程
- 模块
- 协议名
- 串口号
- 波特率
- 收/发方向
- 原始 hex
- 解码摘要
- traceId

没有原始 hex 的日志，后面大概率不够用。

## 5. 建议先准备的调试能力

新壳第一版就留这些入口：

- 串口测试页
- 协议测试页
- GPIO 状态查看页
- 设备信息页
- 日志导出页

当前已经能直接用的是：

- 正式业务首页和调试入口已经分开：
  - 首页走业务态驾驶台
  - 业务模块页和调试页默认收进 debug 环境入口
- 业务首页上不再直接堆整段底座和配置文本
- 联调还是从业务模块页和调试页进，不要回头把调试按钮放回正式首页
- 屏显协议回放
- 808 / AL808 回放
- 全量样例回放
- 打开 / 关闭选中串口
- 连接 / 断开选中 Socket
- 读 / 写选中 GPIO
- 打开 / 关闭选中 Camera
- 读取一次 RFID
- 请求同步系统时间
- 请求重启
- 跑当前业务模块样例
- 跑全部业务模块样例
- 看业务模块状态和最近一次执行结果
- 绑定 / 解绑 GPS 监听
- GPS 绑定默认跟配置里的 `debugReplay.gpsSerialKey` 走
- 选中串口后手工发 HEX
- 选中 Socket 后手工发 HEX
- 解析一段 808 / AL808 HEX
- 看当前选中通道的开关状态
- 看当前底座状态摘要
- 看当前 Socket 协议监听有没有绑定、最近有没有切出完整帧
- 导出调试包
- 日志清空 / 刷新
- 查看当前配置来源和运行期文件位置
- 把运行期配置重置回打包值
- 串口 `stub` 模式下直接带回包 echo 日志
- 串口 `real` 模式下开始记录 `/dev/ttySx` 收发日志
- 串口抽象层已支持注册收包监听，后面 GPS / 外设解析都不用直接改适配器
- Socket 抽象层已支持注册收包监听，后面 JT808 / AL808 回包不再只剩一串原始 hex
- 首页和调试页现在已经共用一套运行时，调试页打开的通道和 GPS 监听状态回到首页就能直接看到
- `protocol` 层已经补 `Jt808FrameStreamParser`，Socket 收到零散字节后能先切成完整帧
- `domain` 层已经补 `Jt808SocketMonitor`，回包会直接输出消息 ID、终端号、流水号和校验摘要
- 808 / AL808 demo 里已带报站、通用应答、违规信息、司机考勤、校时、终端属性、请求消息、升级通知、线路切换样例
- `protocol` 层已经补了 GPS `RMC / GGA` 纯解析器，后面真机只接串口输入
- `protocol` 层已经补 GPS 串流分帧缓冲，后面串口读到原始字节就能直接组句
- `protocol` 层已经补 GPS 串流定位快照合成，后面串口读线程只管喂字节
- 对讲包工厂已单独拆出，后面语音/喊话联调直接接这一层
- 调试包现在会直接导出协议 mock 目录，方便离线核对当前内置样例
- 协议日志现在除了原始 HEX，还会补一行紧凑摘要；JT808 / AL808 会直接给消息 ID、长度、终端号等最小解释

串口测试页至少要能做这些事：

- 选串口
- 选波特率
- 手工发 hex
- 实时看回包
- 保存收发记录

协议测试页至少要能做这些事：

- 输入业务参数
- 生成报文
- 展示 hex
- 用收到的 hex 做反解
- 输入原始 NMEA 语句做 GPS 解析

## 6. 出问题时怎么排

统一按这条线排，不要乱跳：

1. 是不是必现。
2. 是页面问题、业务问题、协议问题，还是设备问题。
3. 有没有完整日志。
4. 有没有原始报文。
5. 同样的输入在 fake 数据下能不能复现。
6. 真机上到底是没发出去、没收到、还是收到了但解错了。

如果已经切到模块层，优先再补一层判断：

7. 是底座问题，还是模块编排问题。

## 7. 每次提需求要顺手更新什么

每次做需求，至少补这三样：

1. 变更记录
2. 涉及模块
3. 新增或变更的日志点

如果需求碰到下面这些，也要顺手补文档：

- 新串口
- 新 GPIO
- 新协议
- 新调试入口
- 新的终端映射关系
- 新增或修改的配置字段

如果当前不是周末真机时间，优先做：

1. mock 样例补齐
2. HEX explain / inspector
3. 页面和配置持久化

把口位、权限、驱动、时序留给真机阶段单独确认。

## 8. 当前建议先落地的调试清单

第一批先做：

- 统一日志接口
- 协议 hex 格式化工具
- 协议收发 traceId
- DVR 触摸坐标要同时记录原始 View 坐标、换算后的协议坐标、按下/移动/松开状态
- 调试入口占位页
- 屏显 / 808 样例回放
- socket stub 通道

当前 DVR 触摸链路的最低日志要求也固定下来：

1. UI 日志必须同时看到 `phase`、原始 View 坐标、View 宽高、换算后的协议坐标。
2. 协议发送日志必须同时看到 `DVR_TOUCH_EVENT`、HEX、phase、x、y、checksum。
3. 真机联调时如果只知道“点了没反应”，先回看这两层日志，先判断是没采到触摸、坐标换算错误，还是串口发出后设备无响应。

建议周末现场按这个顺序看：

1. 打开视频监控页，确认 Camera 已打开。
2. 在预览区左上、中心、右下各点一次，记录 UI 日志里的 raw 和 scaled。
3. 对照协议日志确认三次都发出了 `DVR_TOUCH_EVENT`，且 x/y 有明显变化。
4. 如果 DVR 无响应，再判断是坐标系、字节序还是接线，不要先改页面结构。

第二批再做：

- 报文回放
- 问题日志打包导出
- 假设备适配器
- 单协议 replay 测试

## 9. 配置怎么改

后面需要改口位、波特率、host、port 时，统一走下面顺序：

1. 先改 `config/terminal-config.local.json` 或模板文件
2. 设备联调时优先改运行期文件 `terminal-config.runtime.json`
3. 再补 `docs/配置管理.md`
4. 最后才改真机适配代码

不要再出现：

- 为了联调临时改页面代码
- 为了换 host 直接改 Java 常量
- 为了切真连接直接改 adapter 实现
- 串口和 socket 两边各放一份值
- 设备上明明能改 runtime 配置，却还回头去改 assets

## 10. 当前调试页能直接干什么

现在这页已经能直接做：

- 重新加载配置
- 重置运行期配置
- 清空当前内存日志
- 导出调试包
- 回放屏显样例
- 回放 808 / AL808 样例
- 打开 / 关闭选中串口
- 连接 / 断开选中 Socket
- 读取 / 写入选中 GPIO
- 打开 / 关闭选中 Camera
- 读取一次 RFID
- 请求同步系统时间
- 请求重启
- 选中业务模块后执行单模块样例
- 一次性执行全部业务模块样例
- 手工输入 HEX 发到选中的串口
- 手工输入 HEX 发到选中的 Socket
- 手工输入 HEX 做 808 / AL808 最小反解
- 手工输入原始 NMEA 做 GPS 解析
- 看到当前配置从哪来、运行期文件在哪
- 看到当前选中的串口 / Socket 是否已连通
- 看到当前底座状态和设备能力摘要
- 看到当前业务模块状态摘要
- 看到当前 GPS 监听绑在哪个口、最新定位有没有出来
- 导出的调试包里直接带出 GPS 监听状态
- 看到当前 Socket 协议监听绑在哪条通道、最近一帧解析到了什么
- 导出的调试包里直接带出 Socket 协议监听状态
- 导出的调试包里直接带出底座状态、业务模块状态和增强版终端自检

所以后面真机联调，先别急着改业务页，优先用这页做口位、host、报文验证。

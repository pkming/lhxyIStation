## 1. 当前路线

### 1.1 已定路线

#### 1.1.1

当前路线已经定了，不再反复摇摆：

1. 正式 UI 以旧 `4.4` 项目为唯一基线
2. 新项目继续保留现在的新架构
3. debug 页面继续保留，但只走 debug 路径
4. 后面做法是：
   - 先还原旧 UI 壳
   - 再接新逻辑
   - 再对照旧逻辑补齐
   - 协议和联调最后单独算

#### 1.1.2 对照原则

这里的“对照旧项目”不是照抄旧代码实现，而是把边界定清楚：

1. 要对照的是旧项目的功能结果、页面行为、保存语义、前置校验、联动关系。
2. 不要求继承旧项目里重复、耦合、双写、硬编码、历史包袱式实现。
3. 新架构下优先保留统一配置、统一运行时、统一设备接口，不回退到旧项目那种散落式状态保存。
4. 如果旧项目行为明显不合理，不直接照搬；要先在代码或文档里注明“这是旧行为，但本次不继承”。
5. 如果只是先做了一半对照，也要明确写成“字段已对齐 / 行为未对齐”，不能口头默认已经完成。
6. 当前阶段核心目标是“快速复刻旧项目功能到新壳”，不是先把旧项目所有实现细节研究透。
7. 文档默认先写到“承接位置”这一层，方法名只给关键行为补，不把文档写成方法索引表。

#### 1.1.3 兼容方向约束

后面会话里如果再讨论“兼容”怎么做，统一按下面判断：

1. 默认是新项目去兼容旧 `4.4 / K80` 的业务结果。
2. 不是旧 `4.4` 去向前兼容新项目的新做法。
3. `Android 11 / M90` 是目标平台，不是第二套业务标准。
4. 兼容不能把系统重新做回复杂、双写、散落式结构。

所以后面任何方案如果出现下面任一情况，都先视为方向有问题：

1. 为了兼容，新增一套并行业务实现。
2. 为了兼容，把逻辑重新塞回 Activity 或页面层。
3. 为了兼容，大量增加特判、双状态源、双保存路径。

建议后面继续沿这个标准写注释：

1. 注释只写在迁移决策点。
2. 注释优先说明“旧行为是什么，新壳为什么这么接”。
3. 不给纯直译代码动作写注释，避免把文件变成流水账。

当前推进时还要优先遵守一个顺序：

1. 先补旧页面入口和主流程。
2. 再补用户能感知到的按钮、反馈、保存语义。
3. 最后再继续下沉到实现细节和结构收口。

周末真机前，基础设置 / 站点学习 / 视频监控-DVR 统一先看 `周末调试前核对单.md`，
先区分“已接能力”和“已知未完成项”，再上手调试。

3. 语音通话

系统设置这块当前已经不再只是零散页面逻辑，而是开始走统一配置层：

1. `f_network` 现在已经能把 Socket host / port 和默认调度通道写回运行期配置
2. `f_serial_port` 现在已经能把首批串口波特率和协议选择写回运行期配置
3. `f_newspaper_setup / f_tts_setup / f_language / f_other_setup` 现在已经开始读写统一 `basicSetup` 配置段

注意：

1. 这里的“已接通”默认只代表新架构路径已经成立，不自动等于旧项目完整业务逻辑已经 1:1 对齐。
2. 当前系统设置已经补回第一批旧行为语义：确认弹窗、部分保存后重启、网络/串口互斥方向的一半、无线页跳系统设置。
3. 当前仍未完全补齐的旧行为要继续显式记录，尤其是“资源导入前置判断”和“完整互斥闭环”。

#### 3.3.1

目前四类页面已经能切旧布局，其中四类都已经开始挂回首批真实逻辑：

   - `f_attendance` 已接签到模块状态和签到/签退动作

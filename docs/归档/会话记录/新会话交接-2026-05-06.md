## 2. 当前明确不要误判的点

### 2.1 不要把控件实现差异当成真机阻塞

当前已经恢复的是滚轮式选择交互，不再是列表式 `AlertDialog`。

但还没有恢复的是旧项目原始 `WheelPicker` 控件本体，所以如果真机上看到“能滚动、能确认，但视觉和阻尼不完全一样”，先归到 UI 细节差异，不要直接判成主链未通。

### 2.2 不要把编译通过当成真机行为通过

这一轮修的是：

1. 导航链
2. Fragment 复用
3. 选择器交互层

这三类当前都已经过了窄编译，但还没做设备上的完整点击回归。

## 3. 下一步先做什么

### 3.1 推荐直接做一轮页面点击回归

下一步最合适的不是继续扩页面，而是把刚收敛的这组页面跑一遍点击级核对。

优先顺序：

1. 首页 -> 登录 -> 菜单 -> 子页 -> 首页回收链
2. 线路选择里的线路/方向切换和确认写回
3. 站点学习里的线路/方向/站点或属性切换、保存推进、末项切上下行
4. 调度中心里的签到/调度两个标签来回切换后的状态保留

原因：

1. 这轮刚修的都是“能编译，但最容易在点击层面露出细节问题”的地方。
2. 再继续堆功能前，先把这组主导航页跑稳更能避免返工。

# 项目架构

## 1. 这个新项目要解决什么问题

旧项目能跑，但问题也很明显：

- 硬件接入、业务流程、页面逻辑全缠在一起。
- 串口、GPIO、Camera、RFID 都是旧平台写法。
- 协议代码和发送通道耦合太紧，不方便复用。
- 出问题时不好定位，日志也不够成体系。

新项目的目标不是“换个目录继续堆代码”，而是先把下面几件事做好：

- `架构清楚`
- `模块归类清楚`
- `协议只管编解码，不直接碰 UI 和硬件`
- `硬件都走抽象层`
- `日志能跟着一条业务链跑到底`
- `调试工具单独封装，不要散在页面里`

## 2. 目录结构

```text
IStationDevice-Android11/
├── app/
├── config/
├── docs/
├── modules/
│   ├── core/
│   ├── domain/
│   ├── protocol/
│   ├── device-api/
│   ├── device-m90/
│   ├── runtime/
│   └── debug-tools/
└── scripts/
```

## 3. 模块怎么分

### 3.1 `app`

只管这些：

- 应用启动
- 页面导航
- 正式业务首页
- 业务模块页
- 业务模块操作台
- 页面状态展示
- debug 调试入口
- 依赖组装

这里不要直接写：

- 串口打开
- GPIO 读写
- 协议打包
- Camera ID 判断

当前首页职责已经开始收口成两层：

- 正式路径走业务首页
- 联调路径走业务模块页和调试页

后面保持这个原则：

- 业务首页对齐旧终端主界面
- 模块页和调试页只在 debug 环境保留快捷入口

### 3.2 `modules/core`

放项目最底层的公共能力：

- 日志框架
- 错误模型
- 时间、线程、配置读取
- 十六进制工具
- 基础 Result/Event 模型

这个模块不懂业务，也不懂 M90。

### 3.2.1 配置文件单独管理

新壳里这些值统一不再散在代码里：

- 串口口位
- 波特率
- 串口模式
- 调度 host / port
- GPIO 路径
- Camera 通道映射
- RFID 文件 / 命令桥接
- system ops 命令模板
- 调试页默认回放入口
- 默认 GPS 串口入口

统一走：

- 根目录 `config/`
- 构建时打包到 `assets/terminal-config.json`
- 运行时由配置仓库统一读取

这样后面开发、测试、运维都能对着一份配置说话。

### 3.3 `modules/domain`

放业务编排：

- 终端配置模型
- 线路/车辆/司机等业务模型
- 业务用例
- 任务编排
- 页面和协议之间的业务规则

这里可以懂“签到、切线、报站、升级”这些词，但不应该直接碰串口和 GPIO。

当前先把业务模块编排中心立在这里：

- `dispatch`
- `station`
- `signin`
- `camera-dvr`
- `upgrade`
- `file`

统一由 `TerminalModuleHub` 托管：

- 模块状态
- 模块样例动作
- 模块扩展动作
- 首页摘要
- 调试页模块入口
- 自检和导出里的模块状态

页面层后面统一按这个规则走：

- 页面只负责触发 `moduleKey + actionKey`
- 模块自己保存最近一次业务状态和动作结果
- 设备调用、协议回放、调试包导出都优先收进模块，不再散到页面里

三条主业务链现在额外收了一层状态对象：

- `DispatchState`
  - 调度消息
  - 班次号
  - 计划发车/到站时间
  - 是否参加运营
- `StationState`
  - 线路
  - 上下行
  - 本站/下站/终点
  - GPS 口和最新定位
- `SignInState`
  - 司机名
  - 卡号
  - 上班/下班签到状态

首页后面也直接读这三份状态：

- 调度卡片不再从模块摘要里截字符串
- 报站卡片直接显示线路、本站/下站、阶段和定位
- 签到卡片直接显示司机、卡号和签到/签退结果

页面层除了首页和 debug 页，现在还补了一层正式业务页：

- `ModuleCenterActivity`
  - 首页六个按钮都跳到这里
  - 每个模块各自带状态区和正式业务动作
  - 这样首页不再承担所有业务按钮细节
  - 当前已补第一批正式动作：
    - 调度：确认调度、执行发车
    - 报站：重复报站、停止报站
    - 签到：手动签退

### 3.4 `modules/protocol`

放各类协议的纯逻辑：

- 808 / AL808 / ALINK
- 485 外设协议
- DVR 控制协议
- DVR 触摸协议
- GPS 数据解析

当前已经落地的第一批迁移是：

- 通达屏显协议
- LHXY 屏显协议
- 恒舞屏显协议
- DVR 键位协议
- DVR 触摸协议第一版
- JT808 / AL808 最小公共打包内核
  - 注册
  - 鉴权
  - 心跳
  - 位置信息汇报
  - 报站信息
  - 通用应答
  - 违规信息
  - 司机考勤
  - 校时应答
  - 查询终端属性应答
  - 请求消息
  - 升级通知
  - 线路切换信息
  - AL808 切换线路应答
- 对讲包协议工具
  - 独立于 JT808
  - 单独放 `protocol/intercom`
- GPS 纯解析器
  - RMC
  - GGA
  - 串流分帧
  - 串流定位快照合成
  - 经纬度原始值和十进制度
  - 海拔和卫星数

配合设备层这边，当前 `socket.mode=real` 时已经能完成：

- 建链
- 发包
- 收包日志记录

串口这边当前也已经按同样思路落成：

- `serial.mode=stub`
- `serial.mode=real`
- `real` 版先支持直接访问 `/dev/ttySx` 做收发基础链路
- 波特率控制后面再接厂商 JNI / SDK
- 串口抽象层已支持收包监听，后面 GPS / 485 / 设备状态都能从这里接上来
- GPS 也先按同样原则做：
  - 解析逻辑先独立进 `protocol`
  - 真串口读取后面挂到 `device-m90`
  - 页面和业务层不直接解析 NMEA 字符串
  - `domain/gps/GpsSerialMonitor` 负责把串口字节转换成定位快照日志
- Socket 这边也按同样原则做：
  - `device-api` 只负责连接、发包、收包监听
  - `protocol/jt808/Jt808FrameStreamParser` 负责串流分帧
  - `domain/socket/Jt808SocketMonitor` 负责把回包切帧并输出解析摘要

协议模块只负责：

- 编码
- 解码
- 校验
- 报文模型
- 协议会话状态

协议模块不要直接做这些：

- 打开串口
- 发 socket
- 操作页面
- 直接写日志文件路径

### 3.5 `modules/device-api`

这里是新项目最关键的边界层，先把接口定住。

建议至少拆出这些接口：

- `SerialPortAdapter`
- `SocketClientAdapter`
- `GpioAdapter`
- `CameraAdapter`
- `RfidAdapter`
- `SystemOps`

业务和页面只依赖这些接口，不依赖具体终端实现。

### 3.6 `modules/device-m90`

这里只放 M90 相关实现：

- 串口口位
- GPIO 映射
- Camera 通道
- RFID SDK/JNI 接入
- 系统能力封装

如果后面还要兼容别的终端，就新增 `device-xxx`，不要继续改业务层。

### 3.7 `modules/debug-tools`

物联项目后期最怕“问题能复现，但看不清”。这个模块要单独留出来。

这里准备放：

- 串口收发测试页
- 十六进制报文发送页
- 协议解码查看页
- 日志导出页
- 报文回放页
- 设备状态页

调试工具继续保留，但默认不作为正式业务入口。
正式版页面只看业务态信息，调试工具从 debug 环境进入。

当前已经先落地的入口是：

- 屏显协议回放
- 808 / AL808 回放
- 全量样例回放
- 打开 / 关闭选中串口
- 连接 / 断开选中 Socket
- 读 / 写选中 GPIO
- 打开 / 关闭选中 Camera
- 读取一次 RFID
- 请求同步系统时间
- 请求重启
- 跑当前业务模块
- 跑全部业务模块
- 业务模块状态展示
- 手工 HEX 发到选中的串口 / Socket
- 808 / AL808 最小 HEX 反解
- GPS NMEA 纯文本解析
- 当前通道状态展示
- 当前底座状态展示
- Socket 协议监听状态展示
- 调试包导出
- 日志清空和刷新
- 配置来源和运行期文件位置展示
- 运行期配置一键重置

调试能力不要散在业务页面里。

### 3.8 `modules/runtime`

这里专门放跨页面共享的运行时对象：

- 共享串口适配器
- 共享 Socket 适配器
- 共享 GPIO 适配器
- 共享 Camera 适配器
- 共享 RFID 适配器
- 共享 GPS 监听器
- 共享 Socket 协议监视器
- 共享系统能力封装
- 共享业务模块编排中心

当前 runtime 已经统一托管：

- 底座适配器
- 协议监听器
- 业务模块 hub
- 当前生效配置

这样首页、调试页、后面正式业务页看到的都是同一套模块状态和设备状态。

这么拆的原因很简单：

- 首页要看当前运行状态
- 调试页要开关串口和 Socket
- 后面业务页也会复用同一套设备连接

如果每个页面自己 new，一定会出现：

- A 页打开了串口，B 页看起来还是没打开
- 调试页绑定了 GPS，首页却看不到状态
- 同一条连接被开成两份
- 调试页改了 GPIO / Camera / RFID 状态，别的页却完全不知道

## 4. 依赖方向

只允许按这个方向依赖：

```text
app
  -> domain
  -> protocol
  -> device-api
  -> runtime
  -> debug-tools

domain
  -> core
  -> protocol
  -> device-api

protocol
  -> core

device-m90
  -> core
  -> device-api

runtime
  -> domain
  -> device-api
  -> device-m90

debug-tools
  -> core
  -> domain
  -> device-api
  -> device-m90
  -> runtime
```

明确禁止：

- `protocol -> app`
- `protocol -> device-m90`
- `domain -> device-m90`
- `app` 里直接写硬件访问细节

## 5. 协议封装怎么做

协议层统一按一套结构来，不要每个协议写法都不一样。

建议每个协议都拆成下面几类对象：

- `xxxFrame`
  - 一帧原始报文
- `xxxMessage`
  - 语义化消息对象
- `xxxCodec`
  - 编码和解码
- `xxxValidator`
  - 长度、校验、帧头帧尾校验
- `xxxSession`
  - 协议上下文、状态、重试、分包拼包
- `xxxDebugFormatter`
  - 调试输出，把报文转成好读的文本

这样后面无论是：

- 串口发
- socket 发
- 文件回放
- 调试页手工发

都可以复用同一套协议逻辑。

旧项目协议迁移时，优先顺序按这个来：

1. 先迁纯报文拼装和校验
2. 再迁协议会话状态
3. 最后再接真实发送通道

## 6. 日志流怎么设计

这个项目后面最关键的不是“有没有日志”，而是`能不能顺着一条链查`。

建议日志分 6 类：

- `UI`
- `BIZ`
- `PROTOCOL_TX`
- `PROTOCOL_RX`
- `DEVICE`
- `ERROR`

每条日志尽量带这些字段：

- 时间
- 线程
- 页面/入口
- 业务动作
- 协议名
- 方向
- 串口号或通道号
- 原始 hex
- 解码摘要
- 关联 id

推荐同一条业务链共享一个 `traceId`，例如：

- 页面点击发起
- 业务用例开始
- 协议组包
- 串口发送
- 设备回包
- 协议解码
- UI 更新

这样后面查问题时不用到处拼。

## 7. 调试能力要提前留口

不要等接硬件的时候再临时加日志和测试按钮。

第一阶段就应该预留：

- 调试开关
- 协议原始报文查看
- 串口口位切换
- 发送测试帧
- 保存收发记录
- 导出问题日志

建议把调试能力分成三层：

1. `协议级`
   - 不连硬件，只测编解码。
2. `适配层级`
   - 用 fake adapter 或 replay 数据跑流程。
3. `设备级`
   - 连 M90 真机做单设备联调。

## 8. 文档更新规则

后面每来一个需求，至少更新下面两个地方：

1. `docs/需求变更记录.md`
2. 受影响的架构章节

如果需求会影响下面这些内容，还要同步更新：

- 模块归属
- 协议封装边界
- 日志字段
- 调试工具入口
- 终端映射说明

目标很简单：后面任何人进项目，先看文档就知道该从哪里找。

## 9. 当前阶段先做什么

第一阶段先做空心壳，不接真实硬件：

- 起 Android 11 新工程
- 搭主界面和菜单骨架
- 把模块边界建起来
- 先把日志框架和调试入口留出来
- 先迁纯逻辑、纯协议代码

暂时不做：

- 串口真机联调
- GPIO 真实控制
- Camera 通道适配
- RFID 接入
- 静默安装、重启、校时

## 10. 当前最重要的四件事

1. `架构先定住`
2. `模块归类要清楚`
3. `协议和发送通道彻底拆开`
4. `日志流和调试入口先留好`

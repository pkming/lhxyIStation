# 工作日联调对照说明

更新时间：2026-04-27

这份文档用于工作日继续联调时直接对照。目标不是重新解释项目，而是把周末已经跑通的内容、还差的内容、现场应该怎么测写清楚。

## 1. 当前结论

### 已经跑通

1. ADB 可用，应用可以安装到真机。
2. GPS 已经从 `/dev/ttyS5 @115200` 收到 NMEA，并能解析出有效定位。
3. 摄像头真机链路已通，Camera2 预览已接入 `LegacyVideoMonitorActivity`。
4. 真机实际 Camera ID 已确认不是旧项目假定的 `4/5/6/7`，而是：
   - `av_out = 100`
   - `reverse = 101`
   - `middle_door = 102`
   - `monitor = 103`
5. 首页 `DVR` 入口已恢复，进入后会打开视频监控页。
6. 视频监控页已补手动调试按钮：
   - `开门` -> `middle_door`
   - `倒车` -> `reverse`
   - `DVR` -> `av_out`
   - `关门` -> 关闭当前视频
7. DVR 页面按键和触摸事件已有发送入口。
8. DVR 串口监听和握手逻辑已有代码入口。

### 还没闭环

1. 开门/关门的自动触发还没闭环。
   - 目前只是手动按钮验证视频通道。
   - 自动触发需要拿到 IO 真实 `valuePath`，或者 DVR 发来的倒车/中门/退出信号帧命令码。
2. DVR 串口还需要现场把配置切到真实 RS232-1/DVR 后验证。
3. DVR 反向业务帧目前只明确处理了握手、考勤应答、调度下发、公告下发。
4. DVR 倒车/中门/退出信号协议还没有明确命令号，工作日需要向设备/DVR 供应商确认或抓原始回包。
5. GPIO 真实模式不能贸然打开，因为当前 `valuePath` 还是空。

## 3. 设备运行配置基线

工作日联调前，先确认设备上的运行配置。

```powershell
adb shell cat /sdcard/Android/data/com.lhxy.istationdevice.android11/files/config/terminal-config.runtime.json
```

摄像头已验证可用的配置应接近：

```json
{
  "camera": {
    "mode": "real",
    "channels": {
      "av_out": {
        "cameraId": "100"
      },
      "reverse": {
        "cameraId": "101"
      },
      "middle_door": {
        "cameraId": "102"
      },
      "monitor": {
        "cameraId": "103"
      }
    }
  },
  "debugReplay": {
    "cameraChannelKey": "av_out"
  },
  "gpio": {
    "mode": "stub"
  }
}
```

注意：`gpio.mode` 先保持 `stub`。不要切 `real`，除非已经拿到每个 IO 的真实 `valuePath`。

如果要测 DVR 串口，再把下面几项切到真实配置：

```json
{
  "serial": {
    "rs232_1": {
      "portName": "ttyS3",
      "baudRate": 9600,
      "mode": "real"
    }
  },
  "basicSetup": {
    "serialSettings": {
      "rs2321Protocol": "DVR"
    },
    "protocolLinkageSettings": {
      "dispatchOwner": "serial_rs232_1"
    }
  }
}
```

如果只是测摄像头，不要急着切 DVR 串口配置。

## 4. 工作日联调顺序

### 第一步：确认设备在线

```powershell
adb devices
```

如果设备不在线，先处理 USB、授权、线材和 ADB 连接，不要继续看业务日志。

### 第二步：确认当前 APK 是新版本

重新安装：

```powershell
$env:JAVA_HOME='C:\Program Files\Android\Android Studio\jbr'
$env:Path = "$env:JAVA_HOME\bin;" + $env:Path
.\gradlew.bat :app:installDebug
```

启动后从首页进入 `DVR`，视频页左侧应看到 `开门 / 倒车 / DVR / 关门` 四个按钮。看不到就说明 APK 不是最新，或者进的不是旧版 `LegacyVideoMonitorActivity`。

### 第三步：验证摄像头通道

打开日志：

```powershell
adb logcat | findstr "LegacyVideoMonitorActivity M90RealCamera CameraDvrBusinessModule"
```

操作顺序：

1. 首页点 `DVR`。
2. 默认应打开 `av_out`。
3. 点 `开门`，看是否切到中门画面。
4. 点 `倒车`，看是否切到倒车画面。
5. 点 `DVR`，看是否切回 DVR/AVOUT。
6. 点 `关门`，看是否关闭当前视频。

预期日志：

```text
camera preview started: av_out -> 100
camera preview started: middle_door -> 102
camera preview started: reverse -> 101
camera closed: ...
```

如果某一路没有画面，但日志显示 `camera preview started`，优先查对应视频线/摄像头本身。

如果日志直接报 `camera preview open failed`，优先查 `cameraId`、权限和 HAL。

### 第四步：确认物理通道映射

现在的软件映射只是当前真机实测可打开，不代表线束物理含义一定最终正确。

现场需要逐路确认：

| 软件通道 | 当前 cameraId | 预期含义 | 现场确认 |
| --- | --- | --- | --- |
| `av_out` | `100` | DVR/AVOUT | 待确认 |
| `reverse` | `101` | 倒车 | 待确认 |
| `middle_door` | `102` | 中门/开门 | 待确认 |
| `monitor` | `103` | 监控 | 待确认 |

确认后再把结果写回配置和文档。

### 第五步：验证 GPS

日志：

```powershell
adb logcat | findstr "GpsSerialMonitor M90RealSerial"
```

预期有效定位日志类似：

```text
GPS:
- valid=true
- latitude=22.714103
- longitude=114.040167
- usedSatellites=7
```

GPS 已经跑通过。工作日只需要确认：

1. 冷启动后能否重新定位。
2. 时间、经纬度、卫星数是否稳定。
3. 业务上报里是否使用同一份 `GpsFixSnapshot`。

### 第六步：验证 DVR 串口握手

先切配置到：

1. `serial.rs232_1.mode = real`
2. `serial.rs232_1.portName = ttyS3`
3. `serial.rs232_1.baudRate = 9600`
4. `basicSetup.serialSettings.rs2321Protocol = DVR`
5. `basicSetup.protocolLinkageSettings.dispatchOwner = serial_rs232_1`

日志：

```powershell
adb logcat | findstr "DvrSerialMonitor DvrSerialDispatch M90RealSerial"
```

预期：

```text
DvrSerialMonitor ... 已绑定 DVR 串口监听: rs232_1/ttyS3
DvrSerialDispatch / M90RealSerial ... 55 01 00 00 56 AA
DvrSerialMonitor ... DVR 在线
```

如果没有 `DVR 在线`：

1. 查 DVR 是否上电。
2. 查 RS232-1 接线是否正确。
3. 查 `ttyS3` 是否真的是 DVR 口。
4. 查 DVR 波特率是否 9600。
5. 查 DVR 是否认可握手 `55 01 00 00 56 AA`。

### 第七步：验证 DVR 按键/触摸

进入 `DVR` 视频页后，点左侧数字键、方向键、`M/ENT/ESC/DEL`。

预期日志：

```text
DvrSerialDispatch ... DVR_KEY_EVENT via rs232_1 -> ...
```

点击视频区域，预期：

```text
LegacyVideoMonitorActivity ... touch phase=down / raw=(x,y) / scaled=(x,y)
DvrSerialDispatch ... DVR_TOUCH_EVENT via rs232_1 -> ...
```

触摸协议当前假设：

1. 坐标系 `1280x800`
2. 格式 `55 AA 05 X高 X低 Y高 Y低 状态 校验`
3. `down/move = pressed`
4. `up/cancel = released`

如果 DVR 触摸位置不对，工作日重点核对：

1. X/Y 是否反了。
2. 是否需要镜像。
3. 坐标系是否不是 `1280x800`。
4. X/Y 字节序是否需要调换。

## 5. 开门/关门逻辑怎么定义

当前已做的是手动调试闭环：

1. `开门`：打开 `middle_door` 通道。
2. `倒车`：打开 `reverse` 通道。
3. `DVR`：打开 `av_out` 通道。
4. `关门`：关闭当前视频。

还没做自动触发，因为缺两个关键信息：

1. IO 路径：
   - `IO1 = GPIO1_D0`，文档里对应倒车。
   - `IO2 = GPIO1_D1`，文档/旧设备里对应中门开。
   - 但 Android 侧还没有真实 `valuePath`，所以不能稳定轮询。
2. DVR 事件帧：
   - 文档说 DVR 会发倒车/中门开/退出信号给终端。
   - 但当前还没有明确命令号和 payload 定义。

工作日要拿到以下任意一种信息，自动触发就能继续接：

1. 供应商给出 DVR 倒车/中门/退出信号协议。
2. 现场触发倒车/开门/关门，用 `DvrSerialMonitor` 抓到原始 RX 帧。
3. 系统给出 GPIO `valuePath`，可以从 Android 直接读 IO1/IO2。

抓到 DVR 原始帧时，重点保留完整日志：

```text
DvrSerialMonitor ... dvr raw ttyS3 packet=... bytes=... hex=...
DvrSerialMonitor ... cmd=0x?? len=...
```

## 7. 当前不要做的事

1. 不要在没有 `valuePath` 的情况下把 `gpio.mode` 切成 `real`。
2. 不要把旧项目里 `4/5/6/7` 的 cameraId 直接写回当前设备。
3. 不要把“有摄像头画面”和“DVR 串口在线”混成一个问题，它们是两条链路。
4. 不要只看页面 toast，要同时看 `M90RealCamera / DvrSerialMonitor / DvrSerialDispatch / M90RealSerial` 日志。
5. 不要在没有原始 RX 帧的情况下硬猜 DVR 倒车/中门事件命令号。

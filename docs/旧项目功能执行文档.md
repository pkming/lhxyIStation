# 旧项目功能执行文档

这份文档只解决一个问题：

- 下一个功能开做前，旧项目到底要先梳理到什么程度
- 梳理完之后，怎么变成可执行任务，而不是一堆零散笔记

---

## 1. 结论

### 1.1 要不要先梳理旧项目

要。

但不是先把旧项目所有代码看完，而是先把旧项目按“功能块”梳理成执行文档。

如果不先做这一步，后面每做一个功能都会反复出现同样的问题：

1. 对照边界不清楚
2. 只对齐了页面，没对齐行为
3. 只对齐了字段，没对齐联动
4. 做到一半才发现旧项目还有前置条件和副作用
5. 不同会话对“已经完成”的理解不一致

### 1.2 不要怎么梳理

不要按下面这些方式启动：

1. 不要按包结构从头读到尾
2. 不要先沉到旧项目重复实现细节里
3. 不要一上来就试图把旧项目所有逻辑搬进新壳
4. 不要只写“页面有这个按钮”这种低价值记录

### 1.3 要怎么梳理

统一按“功能块执行单元”来梳理。

一个执行单元至少回答 8 个问题：

1. 旧项目入口在哪里
2. 旧页面怎么表现
3. 旧功能动作是什么
4. 旧前置条件是什么
5. 旧保存或状态流向哪里
6. 旧副作用和联动是什么
7. 新壳由谁承接
8. 这次做到什么算完成

### 1.4 当前核心目标

当前这份文档服务的核心目标不是“把旧项目研究透”，而是：

`尽快把旧项目功能复刻到新壳里。`

所以文档要优先服务推进速度，而不是追求一次性把所有历史细节写满。

后面判断一条记录写得对不对，优先看它有没有帮助下面这 4 件事：

1. 快速定位旧功能入口
2. 快速找到新壳承接点
3. 快速判断这次先补什么
4. 快速判断后面提测时怎么解释差异

如果一条记录只会增加阅读负担、但不会帮助上面这 4 件事，就先不要写太细。

## 2. 对照边界

### 2.1 必须对照的内容

对照旧项目时，必须优先对齐这些：

1. 功能结果
2. 页面行为
3. 保存语义
4. 前置校验
5. 异常反馈
6. 页面间联动
7. 设备或协议触发时机

### 2.2 不直接继承的内容

旧项目里下面这些不默认继承：

1. 重复逻辑
2. SP 和数据库双写这类历史保存方式
3. 硬编码常量
4. 页面直接调设备
5. 旧项目里明显多余的中间状态
6. 仅为旧机型兼容留下的耦合实现

### 2.3 当前统一原则

一句话：

`功能对齐旧项目，结构保持新架构。`

更具体一点：

1. 页面和交互可以像旧项目
2. 业务编排走新壳 domain
3. 配置统一走 ShellConfig
4. 设备动作统一走 device-api
5. 调试和 mock 能力继续独立保留

## 3. 梳理产物长什么样

### 3.1 每个功能块都必须有一条执行卡片

后面每个功能块都按下面模板写，不能只写自然语言描述。

模板如下：

#### 功能名称

1. 旧项目入口
   - Activity / Fragment / 关键按钮 / 关键弹窗
2. 旧项目真实行为
   - 页面初始化
   - 点击动作
   - 保存动作
   - 异常反馈
3. 旧项目前置条件
   - 资源
   - 权限
   - 配置
   - 登录或线路状态
4. 旧项目状态与持久化
   - SP
   - DB
   - 内存态
   - 设备态
5. 旧项目副作用
   - 重启
   - 串口切换
   - 协议互斥
   - 页面跳转
6. 新壳承接位置
   - Activity
   - Fragment
   - Domain UseCase
   - Config
   - Device API
   - 关键方法名（可选）
7. 当前差距
   - 已对齐
   - 未对齐
   - 明确不继承
8. 本轮执行范围
   - 这次准备补哪些
9. 完成标准
   - 页面
   - 行为
   - 联动
   - 文档
10. 验收证据
   - 旧项目依据
   - 新壳结果
   - 差异说明
   - 未继承项说明

### 3.1.1 粒度要求

这份文档默认粒度统一按下面规则：

1. 必填到“承接位置”这一层。
   - 也就是至少写清楚由哪个 Activity / Fragment / UseCase / Config / Device API 承接。
2. 不要求默认细到每个方法名。
3. 只有关键行为才补方法名。

什么算关键行为：

1. 保存动作
2. 前置校验
3. 设备调用
4. 协议切换
5. 重启、跳转、互斥这类副作用
6. 后面很容易和测试产生争议的逻辑

什么情况不用补方法名：

1. 纯展示绑定
2. 容易重构的内部实现
3. 当前只是过渡写法的临时方法
4. 对功能对齐没有决定意义的细枝末节

### 3.2 注释怎么写

代码里可以写对照注释，但只写在迁移决策点：

1. 旧行为保留了什么
2. 旧行为为什么没有原样继承
3. 新壳这里由哪一层承接
4. 当前只对齐到哪一半

不要写成流水账式注释。

### 3.3 这份文档能不能作为测试对齐依据

可以。

但前提是每个功能块后面都要补“验收证据”，不能只有设计意图和实现说明。

后面如果要拿它和测试对口径，至少要满足：

1. 有旧项目依据
2. 有当前新壳行为结果
3. 有明确的差异说明
4. 有明确写出来的未继承项

这样测试提单、研发回收、再次复核时，讨论的是同一份基线，不是各自凭印象。

### 3.4 每个功能块必须留哪些证据

建议每个功能块至少留下面 4 类证据：

1. 旧项目证据
   - 旧代码入口
   - 旧页面截图
   - 旧录屏或关键交互记录
   - 旧文案或弹窗内容
2. 新壳证据
   - 对应代码位置
   - 页面截图
   - 日志
   - 配置前后差异
3. 差异证据
   - 哪些已对齐
   - 哪些未对齐
   - 哪些故意不继承
4. 验证证据
   - 本地验证方式
   - mock 验证结果
   - 真机待验证项

### 3.5 跟测试对口径时怎么判定

后面出现争议时，统一按下面顺序判定，不直接靠口头争：

1. 先看这份执行文档里的功能卡片
2. 再看旧项目证据
3. 再看当前新壳代码和截图/日志
4. 最后看该差异是不是已经被明确标成“不继承”

如果某条没有写进执行卡片，就默认还不算“已经定义完成”。

### 3.6 哪些情况最适合拿这份文档做证据

这份文档特别适合处理下面几类分歧：

1. 测试说“和旧终端不一致”
2. 研发说“旧项目本来就是这样”
3. 双方对“这是 bug 还是未完成项”理解不一致
4. 双方对“这是故意改动还是漏做”理解不一致

不适合只靠这份文档解决的情况也要明确：

1. 真机硬件时序问题
2. 驱动权限问题
3. 串口口位映射问题
4. 现场环境依赖问题

这些仍然需要日志、抓包、真机记录一起看。

## 4. 推荐执行顺序

### 4.1 第一批先梳理哪些

不建议一次把 25 个 UI 单元全梳完。

先梳理这 5 块，收益最高：

1. 基础设置
2. 站点学习
3. 首页细项
4. 视频监控 / DVR
5. 文件管理

原因：

1. 这几块最容易出现“页面像了，但行为没对上”
2. 这几块和设备、配置、旧状态耦合最重
3. 先梳理清楚，后面推进不会反复返工

### 4.2 第二批再梳理哪些

1. 调度中心
2. 信息浏览
3. 语音通话
4. 系统信息
5. 线路选择

## 5. 当前建议的执行方式

### 5.1 开始下一个功能前的标准动作

以后每次开始一个新功能，统一先做这 4 步：

1. 从旧项目定位入口文件
2. 按执行卡片模板补一条记录
3. 明确本轮只做哪一部分
4. 再开始改新壳代码

这里有一个速度原则：

1. 先把功能块卡片补到“能开工”即可。
2. 不要求开工前就把所有旧实现细节补满。
3. 方法名、截图、日志这些证据，可以在实现中和收尾时继续补齐。

### 5.2 做功能时的标准动作

实现时统一按这个顺序推进：

1. 先补文档卡片
2. 再补新壳模型或配置边界
3. 再补页面行为
4. 再补联动和副作用
5. 最后补注释和变更记录

如果目标是快速复刻，执行顺序还要再加一个判断：

1. 先补“用户能看到、能操作、能感知”的旧行为。
2. 再补内部收口和结构优化。
3. 不要为了先把内部设计做漂亮，拖慢旧功能复刻速度。

但有一个例外：

如果某块旧逻辑后面一定会反复返工，比如配置边界、运行时承接、设备接口归属，这类基础边界还是要先收口。

### 5.3 做完后的标准动作

做完一个功能块后，至少要回填这 3 样：

1. 这次对齐到了哪
2. 这次故意没继承什么旧实现
3. 下一轮还差什么

### 5.4 收尾时必须补的验收材料

如果希望后面能直接拿给测试对口径，每个功能块做完后至少补这 5 样：

1. 旧项目对照点
2. 新壳当前结果
3. 截图或日志位置
4. 未完成项
5. 不继承项

缺其中任何一项，后面都容易变成口头拉扯。

## 6. 第一批功能块执行卡片

### 6.1 基础设置

1. 旧项目入口
   - BasicSetupActivity
   - NewspaperSetupFragment
   - NetworkSetupFragment
   - SerialPortSetupFragment
   - TTSSetupFragment
   - LanguageSetupFragment
   - OtherSetupFragment
   - WirelessSetupFragment
2. 新壳承接位置
   - LegacyBasicSetupActivity
   - LegacyBasicSetupSectionFragment
   - ShellConfig / ShellConfigRepository / ShellRuntime
3. 当前已对齐
   - 页面项和主要字段
   - 第一批确认弹窗语义
   - 第一批保存后重启语义
   - 无线页跳系统设置
   - 网络保存时对 RS232-1 的单向互斥处理
   - 文件管理页已提供统一“报站资源已导入”占位写入，状态已落到运行期配置，基础设置保存入口已接前置判断
   - 网络/串口互斥已增加 dispatchOwner 正式状态，不再只靠页面单向清空
   - 首页和调度模块已开始消费 dispatchOwner，串口调度下不会再继续误报为 socket 主链
4. 当前未完全对齐
   - 真实资源导入链路
   - 互斥状态的业务消费和设备动作仍未完全打通，但调度 / 报站 / 考勤 / 公告回复已开始消费 dispatchOwner 并走 DVR 串口主链
   - 视频监控页 DVR 键位已开始走旧键码映射串口发送，触摸协议也已接第一版；当前剩余缺口主要是触摸坐标系和字节序的真机确认
   - 旧项目里广告授权相关行为
5. 本轮后续优先级
   - 先把占位状态源换成真实导入成功态写回
   - 再把 dispatchOwner 从现有调度 / 报站 / 考勤 / 公告回复继续扩到剩余设备动作
6. 当前可作为验收依据的证据
   - 旧项目对应入口已定位
   - 新壳对应承接代码已定位
   - 文档中已明确“哪些行为已对齐，哪些仍未对齐”
7. 当前还缺的验收证据
   - 旧页面关键截图
   - 新壳页面对照截图
   - 保存前后运行期配置差异样例

### 6.2 站点学习

1. 旧项目入口
   - SiteCollectionActivity
   - SiteCollectionFragment
   - OtherCollectionFragment
2. 新壳承接位置
   - LegacySiteCollectionActivity
   - LegacySiteCollectionSectionFragment
   - StationState 相关状态
3. 当前主要差距
   - 线路数据库未接
   - 真实选择器未接
   - 学习流程仍是占位交互
   - 当前保存动作还是“推进站点状态”，不是真实采集落库
4. 开工前必须先梳理
   - 旧选择流程
   - 旧保存结果写到哪里
   - 与线路 / 方向 / 站序的联动
5. 当前测试口径风险
   - 现在如果直接提测，最容易因为“流程占位”和“真实学习逻辑未接”产生争议
6. 周末调试前要明确的结论
   - 当前已恢复的是页面壳、GPS 展示、保存后推进语义
   - 当前未恢复的是旧项目滚轮选择器和真实采集落库

### 6.3 首页细项

1. 旧项目入口
   - MainActivity
   - 首页固定按钮区
   - 数字键盘区
   - toolbar_state
2. 新壳承接位置
   - LegacyMainActivity
   - MainActivity
   - TerminalModuleHub / 各业务模块状态
3. 当前主要差距
   - 顶部状态栏细节未全对齐
   - 固定按钮细动作未全接
   - 键盘细动作未全接
4. 开工前必须先梳理
   - 每个按钮的旧动作
   - 页面反馈
   - 和报站 / 调度 / 视频的联动
5. 当前测试口径风险
   - 首页最容易因为按钮反馈、状态栏细节、联动副作用缺记录而产生争议

### 6.4 视频监控 / DVR

1. 旧项目入口
   - VideoMonitorActivity
   - DVR 串口和触摸区
2. 新壳承接位置
   - LegacyVideoMonitorActivity
   - CameraAdapter / 串口协议层
3. 当前主要差距
   - DVR 键位协议已接
   - 触摸协议已接第一版
   - 旧页面细行为仍未完全对齐
   - 触摸坐标系和字节序仍待真机确认
4. 开工前必须先梳理
   - 旧串口协议触发点
   - 旧触摸反馈
   - 页面与 Camera 状态联动
5. 当前测试口径风险
   - 这块涉及真机和串口协议，没有日志与录屏时很难仅靠页面现象判断
6. 周末调试前要明确的结论
   - 当前已恢复的是 Camera 正式入口和开关链路、DVR 键位协议、DVR 触摸协议第一版
   - 当前待确认的是触摸坐标换算和字节序
7. 当前日志与判读口径
   - 页面日志现在会输出 `phase`、原始 View 坐标、View 宽高、换算后的协议坐标
   - 协议日志现在会输出 `DVR_TOUCH_EVENT` 的 HEX、phase、x、y、checksum
   - 联调时要先看这两层日志，再决定问题是在触摸采集、坐标换算、模块发送还是设备侧

### 6.5 文件管理

1. 旧项目入口
   - FileManageActivity
   - 导入 / 导出 / 升级 / 导日志
2. 新壳承接位置
   - LegacyFileManageActivity
   - FileBusinessModule
   - StationResourceArchiveUseCase
   - 相关 domain / systemOps / export 能力
3. 当前主要差距
   - 导入已支持扫描旧目录 BusRes/BusImport 和 app 专用 imports 目录，并把 ZIP 资源解到运行目录
   - 导出已支持把当前已导入资源重新打包导出
   - 旧文件流程未完整对齐
   - U 盘与升级链路未完整对齐
4. 开工前必须先梳理
   - 旧导入导出前置条件
   - 成功失败提示
   - 升级后动作和权限要求
5. 当前测试口径风险
   - 如果不提前写清“权限 / U 盘 / 升级命令”的边界，后期很容易把环境问题误判成功能 bug

## 7. 这份文档怎么用

### 7.1 使用规则

以后每开始一个功能块，先来这里补卡片，再动代码。

### 7.2 更新规则

每次更新时，至少改这 2 部分：

1. 该功能块的“当前差距”
2. 该功能块的“本轮执行范围”或“后续优先级”

### 7.3 完成判定

一个功能块只有同时满足下面 4 条，才算阶段完成：

1. 页面结构对齐
2. 主要行为对齐
3. 关键联动已说明或已落地
4. 文档里明确写清楚未继承项

---

## 8. 当前建议

下一个功能开始前，先不要再临时口头梳理旧项目。

直接按这份文档挑一个功能块开卡即可。

当前最值得先开卡继续深挖的是：

1. 基础设置里的资源导入前置与协议互斥闭环
2. 站点学习的真实流程
3. 视频监控 / DVR 的旧行为拆解

### 8.1 快速复刻模式下的优先级

如果后面阶段目标明确是“尽快复刻旧项目”，优先级统一按下面排：

1. 先补旧页面入口和主流程
2. 再补关键按钮和页面反馈
3. 再补保存语义和前置校验
4. 再补联动、副作用和设备触发
5. 最后再补实现细节清理和结构优化

也就是说：

先让功能“像旧项目、能走通、能解释”，再继续把内部做稳。

## 9. 提测与争议处理模板

### 9.1 提测时建议附带的最小模板

后面提测一个功能块时，建议直接附下面这 6 行：

1. 旧项目基线：
2. 新壳承接代码：
3. 本轮已对齐：
4. 本轮未对齐：

# 离线推进与协议 Mock 方案

这份文档专门解决两个问题：

1. 平时不能真机时，哪些工作应该继续推进。
2. 新协议文档怎么沉淀成可复用的 mock 资产，而不是每次临时拼 HEX。

---

## 1. 工作日和周末怎么分

### 1.1 工作日

工作日统一只做离线可闭环的事：

1. 页面骨架和旧行为接入
2. 业务状态机和模块动作
3. 配置持久化和运行期刷新
4. 协议模型、编码器、解码器
5. mock 样例和调试页入口
6. 日志、摘要、自检、导出

### 1.2 周末真机

周末真机只做必须上机才能确认的事：

1. 串口口位
2. Socket 连通性
3. 权限和厂商命令
4. 收发时序
5. 驱动差异
6. 真报文和 mock 报文差异核对

---

## 2. 协议文档怎么落成 mock

### 2.1 统一落点

当前统一落到两层：

1. `modules/protocol`
   - 负责协议模型、编码、最小反解、样例生成
2. `modules/domain`
   - 负责 replay 用例和业务回放入口

### 2.2 当前已经固化的 mock 目录

当前代码里已经有统一目录导出：

1. 旧屏显完整样例
2. 旧屏显报站样例
3. JT808/AL808 完整样例
4. 调度主链样例
5. 签到样例
6. 升级样例

调试包会直接导出 `protocol/mock-catalog.txt`，方便离线核对。

### 2.3 每个新协议最少补三样

1. 一条标准成功样例
2. 一条可读摘要或 inspector
3. 一条 replay 入口

这样后面协议文档不再只是 PDF 或截图，而是能直接进代码、进日志、进调试页。

---

## 3. 原始 HEX 怎么看

### 3.1 当前已经有的能力

1. `PROTOCOL_TX / PROTOCOL_RX` 里保留原始 HEX
2. JT808 / AL808 已支持最小反解
3. 调试页可手工输入 HEX 做解析
4. Socket 协议监听会输出最近一帧摘要

### 3.2 当前还没全做完的

1. 不是所有协议都已经有字段级 explain
2. 旧屏显、DVR、其他自定义协议后面还要继续补 inspector

---

## 4. 本轮落地标准

本轮开始按下面这条标准推进：

1. 新协议文档先落协议模型
2. 再补 mock 样例
3. 再补调试页或日志 explain
4. 最后才等周末真机确认

后面不要再出现：

1. 只有真机才能推进
2. 只有一串裸 HEX 没有可读摘要
3. 协议文档看完了但仓库里没有样例资产

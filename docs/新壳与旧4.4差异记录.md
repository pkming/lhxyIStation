# 新壳与旧 4.4 差异记录

这份文档只回答一件事：

- Android 11 新壳现在和旧 `4.4` 项目，到底差在哪

后面如果再讨论“要不要继续对齐旧首页”“哪些行为还没追平”，先看这份，不用每次重新翻代码。

---

## 一句话结论

- 代码结构已经不是旧项目那种大一统写法了，这个差异是刻意保留的。
- UI 已经进入旧页面回迁阶段，但还不是旧 `4.4` 终端的 `1:1` 完整还原。
- 业务动作已经开始往旧终端主流程收，但很多深层行为还没完全追平。
- 后面更合理的做法是：`代码结构保持新架构，UI 和业务行为继续向旧终端收敛。`

---

## 1. 代码差异

### 旧项目怎么写

旧项目的核心是一个很重的 `MainActivity`。

- 首页 UI、按钮事件、串口、GPS、TTS、GPIO、摄像头、服务启动、广播注册都揉在一起。
- 启动时直接开口位、开服务、绑广播。
- 业务动作很多是页面层直接调底层或直接发协议。

当前最能代表旧写法的地方：

- `app/src/main/java/com/lhxy/istationdevice/ui/activity/MainActivity.java`
- `onCreate()` 里直接做权限、界面、网络、TTS、喇叭、摄像头 GPIO、`/dev/ttyS2`、`RS485` 初始化。
- `initService()` 里直接绑网络广播、广告服务、资源下载、升级服务。

这种写法的好处是“都在一个地方”，坏处也很明显：

- 很难拆模块
- 很难替换硬件
- 很难单独调一个业务
- 页面和底层耦合太深

### 新壳现在怎么写

新壳已经切成模块化结构，不再让一个页面自己承接所有业务。

- `LegacyMainActivity`
  - 现在是正式旧首页壳
  - 正式路径从这里进
- `MainActivity`
  - 现在是过渡业务总览页
  - 主要负责总览、状态展示、跳转
- `ModuleCenterActivity`
  - 现在是过渡业务页
  - 每个模块有自己的状态区和动作区
- `TerminalModuleHub`
  - 统一托管六个业务模块
  - 统一分发 `moduleKey + actionKey`
- `DispatchState / StationState / SignInState`
  - 把三条主链单独变成状态对象
- `device-api / device-m90 / runtime / protocol`
  - 把设备抽象、M90 适配、共享运行时、协议纯逻辑分开

一句话说，新壳现在是：

- 页面只负责触发动作和展示状态
- 模块自己保存业务状态
- 设备层和协议层单独放

### 代码层现在已经对齐到哪里

已经对齐的：

- 六个业务模块已经挂起来
  - `dispatch`
  - `station`
  - `signin`
  - `camera-dvr`
  - `upgrade`
  - `file`
- 正式旧首页、过渡业务页、调试页都已经不再直接乱调底层
- 调度 / 报站 / 签到已经有单独状态对象
- 业务动作已经开始通过模块 action 收口

还没完全对齐的：

- 旧项目里很多“页面里顺手做掉”的隐式逻辑
- 自动 GPS 报站完整状态机
- 调度下发后的细分动作和状态迁移
- DVR 串口、触摸、视频联动细节
- 升级、导入导出、系统服务链的老行为
- 菜单页、登录流、顶部状态栏、数字键盘、弹窗这些页面外 UI 单元

所以代码层的判断很明确：

- `结构上不对齐`
- `动作入口已经开始对齐`
- `深层行为还没完全对齐`

## 2. UI 差异

### 旧首页长什么样

旧 `4.4` 首页是典型车载终端布局，不是卡片式页面。

它的结构大概是：

- 顶部一条黑色状态栏
  - 时间
  - 标题
  - 卫星
  - LAN
  - CMS
  - DVR
  - 4G
  - WiFi
  - GPS
- 左侧一整列固定操作按钮
  - 报站
  - 线路
  - 切换
  - 键盘
  - 停止
  - 报警开
  - 菜单
- 中间和右侧是固定业务区
  - 线路
  - 司机
  - 发车时间
  - 速度
  - 数字键盘区

旧首页的特点是：

- 固定分区
- 固定按钮位
- 操作路径短
- 更像设备终端，不像普通 Android App

### 新壳现在的 UI 是什么样

新壳现在已经不是最早的调试摘要页了，但也还不是旧首页翻版。

现在的正式路径是：

- `LegacyMainActivity`
  - 旧版首页骨架
- `LegacyLoginActivity`
  - 旧版登录页骨架
- `LegacyMenuActivity`
  - 旧版菜单页骨架
- `LegacyLineChoiceActivity / LegacySiteCollectionActivity / LegacyFileManageActivity`
  - 第一批菜单子页面骨架
- `LegacyBasicSetupActivity / LegacyVoiceCallActivity / LegacyDispatchCenterActivity`
  - 第一批业务子页面骨架
- `LegacyInfoBrowsActivity / LegacySystemInfoActivity / LegacyVideoMonitorActivity`
  - 第一批信息和系统页骨架
- `MainActivity`
  - 新壳业务总览页
- `ModuleCenterActivity`
  - 新壳模块业务页
- `DebugToolsActivity`
  - 调试页
  - 保留联调用

新壳 UI 的特点是：

- 旧版首页 / 登录 / 菜单三页已经开始回迁
- 菜单第一批八个子页面骨架已经开始回迁
- 新壳业务总览页和模块页还作为过渡页保留
- debug 入口开始和正式路径分开

### UI 现在和旧首页差多少

差异还比较明显。

已经靠近旧终端思路的地方：

- 首页已经变成业务态入口，不再是开发工具页
- 调度 / 报站 / 签到已经开始直接展示真实业务状态
- 正式业务动作已经从首页拆到单独业务页

还没对齐的地方：

- 顶部黑色状态栏没有按旧终端样式还原
- 左侧固定按钮列没有还原
- 数字键盘区没有还原
- 首页 -> 登录 -> 菜单 -> 第一批子页面 这条主导航已经恢复，但不是全部子链都已补齐
- 菜单页虽然已经能进第一批子页面，但很多子页面内部细项、Fragment 和弹窗还没完全还原
- 多个关键弹窗和选择器还没作为正式 UI 单元回补
- 主业务区的布局节奏和旧终端不同
- 视觉上还是“新壳业务页”，不是“旧终端翻版”

所以 UI 层的结论也很直接：

- `现在已经不是调试页`
- `但也还不是旧首页`

## 3. 当前状态判断

### 已经对齐的

- 项目已经有正式旧首页、过渡业务页、调试页三层结构
- 页面层已经开始只做展示和跳转
- 六个业务模块已经有固定入口
- 调度 / 报站 / 签到已经开始读真实状态对象
- debug 入口已经开始收口，不再直接挂在正式路径上

### 半对齐的

- 调度模块
  - 已有加入运营、退出运营、确认调度、发车
  - 但还不是旧项目那种完整调度流
- 报站模块
  - 已有推进一站、重复报站、停止报站
  - 但还不是完整自动报站状态机
- 签到模块
  - 已有读卡、签到/签退状态、手动签退
  - 但还不是完整司机业务闭环
- 首页
  - 已经是业务首页
  - 但布局和节奏还没回到旧终端样式

### 还没对齐的

- 旧首页固定布局和按钮位
- 旧项目大量隐式业务行为
- DVR 相关业务
- 升级、导入导出完整流程
- 所有依赖真机协议和联调确认的部分

---

## 4. 后面怎么处理更合适

代码层建议保持现在这条路，不要回退成旧项目写法。

- 新壳的模块化、状态对象、设备抽象、协议分层都应该保留
- 后面要补的是业务行为，不是把代码再揉回一个大 `MainActivity`

UI 层则应该继续往旧终端靠。

- 首页布局可以继续向旧 `act_main.xml` 收
- 顶部状态栏、左侧操作位、固定业务区可以逐步回补
- debug 能力继续保留，但只放 debug 环境

协议和联调单独算。

- 代码侧可以继续补动作和状态
- 真正 `1:1` 追平旧终端，还得靠协议接入和真机联调
